2026-04-08 账号池 / 网关控制台 UI 调研结论

一、结论

到 2026-04-08 为止，公开资料里并没有一个单独叫“账号池 UI 标准”的官方规范。
但是从以下几类官方产品和官方设计系统中，可以提炼出稳定共性：

1. 身份 / 用户管理台
2. Token / API Key / 授权管理台
3. 云控制台里的资源清单页

这些页面的共同规律不是“首页堆很多展示卡片”，而是：

1. 接入和清单分开，但放在同一个工作区连续完成
2. 一级主操作只能有一个，通常是 Connect / Add / Create
3. 主体必须是 inventory list，不是满屏卡片，也不是长表单
4. 高级项默认折叠，按需展开
5. 响应式不是简单缩放，而是 reposition / resize / reflow / collapse
6. 空状态只能保留一个最重要动作，不能同屏出现多个抢主次的空状态
7. 生成式链接是主流做法，授权链接由后端拼好，前端负责展示 / 打开 / 复制

其中第 7 条与当前 OpenAI 登录最相关：
参考实现并不是先弹工具框让用户继续操作，而是点击登录后，后端直接返回完整 authUrl，前端直接展示、打开、复制，并在失败时提供手动粘贴回调作为兜底。


二、别人的成熟做法

适合我们直接借鉴的结构如下。

1. 顶部区
只放页面标题、环境状态、最后更新时间。
不放巨大展示型英雄卡，不抢首屏操作。

2. 第一工作区：接入区
只放一个一级 CTA，例如“连接 OpenAI”。
点击后直接生成完整授权链接。
紧接着显示：
- 打开授权页
- 复制登录链接
- 当前回调地址

再往下放两个次级入口：
- 粘贴回调
- 文件导入

3. 第二工作区：账号池清单
账号池主体必须是 inventory layout。
每一行只保留高价值字段：
- 账号名
- 模型
- 路由
- 健康
- 配额
- 凭证状态

更多细节不要塞进首屏主清单，而是使用行展开、抽屉、详情页。

4. 第三工作区：运行细节
租约、任务、告警、缓存、拓扑都属于次级信息。
默认不应该抢占第一屏。
应当折叠或放在更下方的分区中。


三、对当前项目的直接要求

当前项目之前的问题主要有：

1. 登录入口工具化
点击登录后出现一个“小弹层”，用户还要在里面继续做一次选择。
这不符合成熟控制台的主流程设计。

2. 账号区布局失衡
右侧使用横向挤压表格，导致内容压缩、阅读成本高、窗口变化时适配差。

3. 首屏主次混乱
概览、状态、接入、运行细节同时争夺注意力。
用户进入页面后不能立刻判断“第一步该做什么”。

4. 不同窗口尺寸适配不足
宽屏、中屏、窄屏下没有明确的布局策略。
页面更多是缩小，而不是重排。


四、我们应该采用的最终结构

第一屏只保留两块：

1. 左侧或上方：接入区
- 连接 OpenAI
- 复制登录链接
- 粘贴回调补导入
- 文件导入

2. 右侧或下方：账号池清单
- 账号列表
- 状态芯片
- 指标条

其他内容全部下沉：
- 缓存命中
- 租约
- 浏览器任务
- 告警
- 拓扑


五、响应式规则

必须明确做三档断点，而不是任意缩放。

1. 窄屏：<= 768
- 单列布局
- 接入区在上
- 账号清单在下
- 禁止主区域横向滚动

2. 中屏：769 - 1279
- 上下两段式布局
- 接入区横向拉宽
- 账号清单改为紧凑列表

3. 宽屏：>= 1280
- 双栏布局
- 左侧接入
- 右侧账号池
- 运行细节继续下沉

不允许仅靠缩小字体来适配窗口。
必须通过重排来完成适配。


六、OpenAI 登录的正确行为

点击“OpenAI 登录”之后，前端应该直接输出完整授权链接，而不是只弹一个小框。

正确流程：

1. 点击登录
2. 前端调用登录启动接口
3. 后端返回完整 authUrl
4. 前端把完整 authUrl：
- 直接打开
- 直接展示
- 支持复制
5. 若自动回调失败，再使用“粘贴回调地址”兜底

授权链接参数的核心结构应当类似：

- response_type=code
- client_id=app_EMoamEEZ73f0CkXaXp7hrann
- redirect_uri=...
- scope=openid profile email offline_access api.connectors.read api.connectors.invoke
- code_challenge=...
- code_challenge_method=S256
- id_token_add_organizations=true
- codex_cli_simplified_flow=true
- state=...
- originator=...


七、后续本地重构计划

本轮先不推送远端，只在本地分支 3 的 Docker 环境内迭代。

执行顺序如下：

1. 重做页面信息架构
首屏只保留“接入区 + 账号池清单”。

2. 重做 OpenAI 登录入口
点击后直接在页面内输出完整 authUrl。

3. 去掉工具化小弹层
把回调解析和文件导入收进同一接入区。

4. 重做账号池清单
宽屏为列表卡片，中屏为紧凑列表，窄屏为单列堆叠。

5. 建立三档响应式验收
逐档截图检查：
- 窄屏
- 中屏
- 宽屏

6. 最后才做视觉收敛
结构正确后，再统一：
- 间距
- 标题层级
- 按钮优先级
- 状态标签
- 空状态
- 阴影
- 对比度

九、最终原则

我们不是在做展示页，而是在做“账号池 / 网关控制台”。

因此页面必须满足：

1. 一进入就知道先做什么
2. 一登录就能拿到完整授权链接
3. 一眼就能看懂账号池状态
4. 一换窗口尺寸也不会乱
5. 一切低频细节都应该自动下沉

如果做不到以上 5 条，就不能算完成。
